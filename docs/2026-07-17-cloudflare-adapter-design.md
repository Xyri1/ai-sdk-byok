# Design: Cloudflare Adapter (`@ai-sdk-byok/cloudflare`)

- **Date:** 2026-07-17
- **Status:** Approved design, pending implementation spec (`specs/003-cloudflare-adapter/`)
- **Goal:** Apps built on the AI SDK and hosted on Cloudflare Workers can use `ai-sdk-byok` to let their users bring their own provider API keys.

## Summary

One new workspace package, `packages/cloudflare`, published as `@ai-sdk-byok/cloudflare`, with two exports:

- `d1Adapter(options)` — a `ByokStorageAdapter` backed by Cloudflare D1.
- `kvCredentialCache(options)` — a `CredentialRecordCache` (spec 002 interface) backed by Workers KV.

Credentials are **always encrypted** with AES-256-GCM via WebCrypto before touching D1 or KV. The integrator supplies a 32-byte master key held as a Worker secret or a Secrets Store binding. There is no plaintext mode and no pluggable crypto surface.

## Decisions Made During Brainstorming

1. **Goal is Workers-hosted apps**, not D1 access from arbitrary runtimes. The adapter consumes bindings (`env.DB`, `env.KV_NAMESPACE`), not the Cloudflare HTTP API.
2. **Scope is storage adapter + KV cache together** in one package, so Workers integrators get the complete performant setup (`createByokManager` + `cachedStorage` + D1 + KV) on day one.
3. **Approach A — always-encrypted — was chosen** over plaintext-with-documentation and a pluggable sealer. Rationale:
   - Cloudflare has no managed per-user secret store or KMS. Worker Secrets and Secrets Store hold deploy-time configuration secrets (Secrets Store: account-level, ~100 secrets, deploy-time bindings) and cannot store dynamic per-user data. D1/KV at-rest encryption is transparent to anyone with data access.
   - The platform-idiomatic pattern for this problem is exactly: ciphertext in D1/KV, master key in a Worker secret, AES-GCM via `crypto.subtle` in-Worker.
   - BYOK credentials are other people's money. A plaintext mode would be chosen by default and fragments the package's cross-adapter security story. The Supabase adapter's headline claim — storage compromise does not expose plaintext credentials — holds for this adapter too.
4. This is a deliberate scope change to `AGENTS.md`: "Non-Supabase storage adapters" leaves Out Of Scope, and "Application-side cryptography" is narrowed to exclude the sealed-credential scheme owned by this package.

## Architecture

### Package layout

```
packages/cloudflare/
  src/index.ts        — exports d1Adapter, kvCredentialCache
  src/crypto.ts       — internal seal/unseal (shared by both exports; not public API)
  src/d1-adapter.ts
  src/kv-cache.ts
  src/*.test.ts
  migrations/0001_ai_sdk_byok_init.sql   — wrangler d1 migrations format
  README.md
```

### Public API

```ts
d1Adapter({
  database: env.DB,                      // D1 binding
  encryptionKey: env.BYOK_MASTER_KEY,    // base64 32-byte key, or async getter
}): ByokStorageAdapter

kvCredentialCache({
  namespace: env.BYOK_CACHE,             // KV binding
  encryptionKey: env.BYOK_MASTER_KEY,
  keyPrefix?: string,                    // default 'ai-sdk-byok:'
}): CredentialRecordCache
```

- `encryptionKey: string | (() => string | Promise<string>)`. The string form covers Worker Secrets (`env.BYOK_MASTER_KEY`); the getter form covers Secrets Store (`() => env.STORE.get()`). Resolved once per adapter instance and imported as a non-extractable WebCrypto `CryptoKey`.
- Bindings are typed structurally (internal `D1DatabaseLike` / `KvNamespaceLike` interfaces, public options accept the binding without a runtime dependency on `@cloudflare/workers-types`) — same pattern as the Supabase adapter's `SupabaseClientLike`. `@cloudflare/workers-types` is a dev-only dependency.
- Everything is Edge-native: `crypto.subtle`, `crypto.randomUUID`, `crypto.getRandomValues` — available in Workers and Node 22+, so the "no Node-only top-level imports" constraint holds and unit tests run under plain Vitest.

### Typical integration

```ts
const manager = createByokManager({
  storage: cachedStorage({
    storage: d1Adapter({ database: env.DB, encryptionKey: env.BYOK_MASTER_KEY }),
    cache: kvCredentialCache({ namespace: env.BYOK_CACHE, encryptionKey: env.BYOK_MASTER_KEY }),
    ttlMs: 60_000,
  }),
});
```

## D1 Schema

```sql
CREATE TABLE ai_sdk_byok_keys (
  id TEXT PRIMARY KEY,                    -- UUID, generated by adapter (crypto.randomUUID)
  user_id TEXT NOT NULL,
  provider TEXT NOT NULL,
  label TEXT NOT NULL DEFAULT 'default',
  key_hint TEXT NOT NULL,
  credentials_ciphertext TEXT NOT NULL,   -- sealed format below; never plaintext
  created_at TEXT NOT NULL,               -- ISO-8601 UTC, set by adapter
  updated_at TEXT NOT NULL,
  UNIQUE (user_id, provider, label),
  CHECK (length(user_id) BETWEEN 1 AND 256),
  CHECK (length(provider) BETWEEN 1 AND 128),
  CHECK (length(label) BETWEEN 1 AND 128),
  CHECK (length(key_hint) BETWEEN 1 AND 4)
);
CREATE INDEX ai_sdk_byok_keys_user_list_idx
  ON ai_sdk_byok_keys (user_id, updated_at DESC, created_at DESC);
```

SQLite has no `timestamptz`; the adapter writes `new Date().toISOString()`. ISO-8601 UTC strings sort lexicographically, preserving the required `updatedAt desc, createdAt desc` list ordering.

### Operations

- **`save`** — one atomic upsert (no advisory locks; D1 serializes writes per database):

  ```sql
  INSERT INTO ai_sdk_byok_keys (id, user_id, provider, label, key_hint,
                                credentials_ciphertext, created_at, updated_at)
  VALUES (?, ?, ?, ?, ?, ?, ?, ?)
  ON CONFLICT (user_id, provider, label) DO UPDATE SET
    key_hint = excluded.key_hint,
    credentials_ciphertext = excluded.credentials_ciphertext,
    updated_at = excluded.updated_at
  RETURNING id, user_id, provider, label, key_hint, created_at, updated_at;
  ```

  Rotation preserves `id` and `created_at`, matching Supabase adapter behavior. `RETURNING` never projects the ciphertext column.
- **`list`** — metadata columns only, `WHERE user_id = ? ORDER BY updated_at DESC, created_at DESC`. `credentials_ciphertext` is never in the projection.
- **`get`** — ciphertext by `(user_id, provider, label)` → unseal → strict shape validation → `{ apiKey }` or `null`.
- **`getById`** — full row by `(id, user_id)` → unseal → `StoredKeyCredentialRecord` or `null`.
- **`delete`** — `DELETE WHERE id = ? AND user_id = ?`; idempotent, no error when the row is already gone.

## Sealed Credential Format

```
v1.<base64url(iv)>.<base64url(ciphertext || gcm-tag)>
```

- AES-256-GCM via `crypto.subtle`; fresh random 96-bit IV per write; key imported non-extractable.
- Plaintext is `JSON.stringify({ apiKey })` — the same envelope the Supabase adapter stores in Vault.
- **AAD binds ciphertext to its logical slot:** `userId \0 provider \0 label` for D1 rows. This defeats ciphertext-swap attacks: an attacker with D1 write access who copies a victim's sealed blob into their own row gets a decryption failure. The slot triple is immutable per row (it is the upsert conflict key), so rotation is unaffected.
- The `v1.` prefix is the forward-compatibility seam: future key rotation or algorithm changes introduce `v2.` readers without a data migration.
- Master key requirement: base64-encoded 32 bytes, generated by the integrator (e.g. `openssl rand -base64 32`), stored via `wrangler secret put` or Secrets Store.

## KV Credential Cache

Implements `CredentialRecordCache` exactly, so it plugs into the existing `cachedStorage()` wrapper with zero core changes.

- **KV key:** `${keyPrefix}${sha256hex(userId + '\0' + keyId)}`. KV keys cap at 512 bytes and `userId` alone can be 256 chars (more once encoded); hashing yields a fixed-length, delimiter-safe key and prevents user enumeration by browsing the namespace.
- **Value:** sealed with the same `v1.` scheme, where the plaintext is `{ "record": StoredKeyCredentialRecord, "expiresAt": <epoch ms> }`. `expiresAt` lives inside the ciphertext so expiry is tamper-proof (GCM-authenticated). AAD = `userId \0 keyId`, so a sealed entry cannot be replayed under another cache slot.
- **TTL, two layers** (KV's `expirationTtl` has a 60-second minimum):
  - Physical: `put(..., { expirationTtl: max(60, ceil(ttlMs / 1000)) })` for eventual cleanup.
  - Logical: on read, decrypt and require `expiresAt > Date.now()`; expired entries are misses. Sub-60s `ttlMs` values therefore behave exactly as configured.
- **Failure posture — self-healing, never blocking:** integrity failures on read (wrong master key, tampered value, malformed blob) are treated as a miss with a best-effort `delete` of the bad entry. Combined with `cachedStorage`'s existing error handling, a KV outage or rotated master key degrades to slower D1 reads, never to failed requests.
- **Documented caveat, not solved:** KV is eventually consistent. A `delete` (rotation/revocation) can leave a stale entry readable in other regions until propagation (~60s) plus remaining TTL. This extends the existing threat-model statement that cache invalidation is not transactional. Stale entries remain ciphertext at rest — staleness is an availability-of-old-key issue, not an exposure issue. Guidance: keep `ttlMs` short.

## Capacity and Platform Limits

Verified against current D1/KV documented limits:

| Limit | Our worst case | Headroom |
| --- | --- | --- |
| D1 row size 2 MB | ~11.6 KB (8,192-char `apiKey` sealed + base64, plus ≤600 B metadata) | ~170× |
| SQL statement 100 KB | static SQL with `?` binds | ✔ |
| 100 bound params / 100 columns | 8 / 8 | ✔ |
| 1,000 queries per invocation (paid) | 1 statement per adapter operation | ✔ |
| KV value 25 MB / key 512 B | ~12 KB value / 76 B hashed key (12-char default prefix + 64-char sha256 hex) | ✔ |
| KV 1 write/sec per key | each cache key written at most once per TTL window | ✔ |

**The real ceiling is D1's 10 GB per-database cap (Workers Paid).** At a typical ~1–1.2 KB per record (row + index), that is roughly 8M stored keys ≈ ~3M users at 2–3 keys each; the absolute worst case (every key at the 8 KB max) is ~850K records. Throughput is not the constraint: writes are rare user-initiated events, and hot read load is absorbed by the KV cache.

The schema is deliberately shard-friendly so the escape hatch needs no schema change: every query is keyed by `user_id` first, uniqueness is per-`(user_id, provider, label)`, and IDs are app-generated UUIDs. If a user base outgrows one database, a future `shardedD1Adapter({ databases, hash })` (or integrator-side routing across per-shard `d1Adapter` instances) distributes users across N databases. Cloudflare supports 50K databases per account by default, raisable to millions. Sharding is explicitly out of scope for v1; the ceiling and the path are documented for capacity planning.

## Error Handling

- Every failure surfaces as the existing `AiSdkByokAdapterError` with the operation name (e.g. `"Cloudflare D1 BYOK adapter failed during save"`) and the underlying error preserved as `cause` — same pattern as the Supabase adapter.
- Error messages and cause chains never contain plaintext credentials, key material, ciphertext, or serialized credential input.
- **Config fail-fast:** `encryptionKey` must decode to exactly 32 bytes. The string form is validated synchronously at construction; the getter form on first use. Wrong-size keys produce a clear "expected 32-byte base64 key" error without echoing the value.
- Malformed rows from D1 (wrong shape, unparseable sealed blob) throw rather than silently returning partial data, mirroring the strict parsing posture of the Supabase adapter.

## Testing Strategy

- **Real SQLite behind a fake D1 binding:** a small test shim wraps `better-sqlite3` (dev-only dependency; Node-only utilities are allowed in tests) exposing `prepare().bind().first()/all()/run()`, so the adapter's actual SQL — upsert conflict behavior, `RETURNING`, index ordering — runs against genuine SQLite semantics.
- **In-memory KV fake** implementing `get/put/delete` with `expirationTtl` bookkeeping.
- **Crypto tests:** round-trip; wrong-key rejection; AAD mismatch rejection (swap ciphertexts between two rows and prove decryption fails); unknown version-prefix rejection.
- **Security-invariant tests** mirroring the Supabase suite: `save`/`list` output never contains ciphertext or plaintext; end-to-end proof that returned credentials stay proxy-wrapped against `JSON.stringify`.
- **Follow-up (not v1):** an integration pass under `@cloudflare/vitest-pool-workers` against real workerd, analogous to `docs/integration-testing.md` for Supabase.

## Spec and Docs Impact

- New `specs/003-cloudflare-adapter/` — requirements, plan, tasks, checklist, decisions.
- `AGENTS.md` — remove "Non-Supabase storage adapters" from Out Of Scope; narrow "Application-side cryptography" to exclude the sealed-credential scheme owned by `@ai-sdk-byok/cloudflare`; add the package to Repository Layout, Public API Requirements, and Security Invariants (master-key custody, ciphertext never in metadata or errors).
- `docs/threat-model.md` — new Cloudflare section: protects against D1/KV storage compromise without the master key; does not protect against master key + storage compromise together, nor stale KV entries surviving until propagation + TTL; losing the master key means users re-enter keys (acceptable: API keys are re-enterable; this is not durable-data loss).
- `docs/architecture.md`, root `README.md` + `README.zh-CN.md`, package README with the Workers quickstart (generate key → `wrangler secret put` → apply migration → compose with `cachedStorage`).

## Out of Scope (v1)

- Sharded / multi-database D1 routing (`shardedD1Adapter`) — documented capacity path, not built.
- Master-key rotation tooling (the `v1.` format prefix reserves the seam).
- Cloudflare HTTP API access to D1 from non-Workers runtimes.
- Durable Objects or R2 backends.
- An `examples/` Worker app (README quickstart only; example app is a candidate follow-up).
